## 0. How to use these docs (read this first, every session)

**Standing directive for every agent/contributor working on this repo,
present and future: minimize token/context usage on every task.**

1. **Read this file's index (§2) to find the right `docs/systems/*/README.md`
   for the system you're touching — go straight there.** Do NOT re-scan the
   repo or read unrelated systems' files.
2. **Read the system's README.md (and its per-script `.md` if one exists)
   before opening any actual `.gd` source.** These docs are written to
   answer purpose/responsibilities/public API/signals/persistence/common
   edits/forbidden edits/tradeoffs WITHOUT needing the source. If a doc
   answers your question, don't also open the source "just to check."
3. **Only open real source code for the specific function's actual
   behavior/logic** — the one thing the docs deliberately don't capture.
   Open only the specific file(s) involved, not the whole folder.
4. **Consult `architecture.json`** (repo root, generated by
   `tools/gen_architecture.py`) for a system's dependency edges, signals,
   and public function signatures before opening source — regenerate it
   with `python3 tools/gen_architecture.py` at the start and end of a
   session (see §5).
5. **If a doc is missing detail you needed and had to read source to find,
   that's a doc bug — add the missing detail back into the relevant
   README.md in the SAME commit**, so the next session doesn't pay that
   same cost again. Doc drift is treated as a bug, same weight as a code bug.
6. **A system with no `docs/systems/*/README.md` yet** (see §2's "pending"
   rows) means it hasn't been migrated to this format. The FIRST nontrivial
   change to such a system must create its README.md as part of that same
   change — this is how migration happens incrementally, not as separate
   follow-up work.
7. Before reporting any fix as done, run `tools/godot_check.sh` (headless
   Godot compile check — see §16) — catches parse/type/autoload errors
   before Brannon has to pull and hit them in his own editor.

This file (§1, §3, §4 below) stays long-form for the game's vision/direction
and repo-wide facts that don't belong to any one system. Everything else has
moved to `docs/systems/*/README.md` — do not duplicate migrated content back
into this file.

## 1. Game vision & direction
Bunker survival/design game, two phases:
- **Pre-Apocalypse** — spend cash, build/buy rooms, wire power, stock supplies.
- **Post-Apocalypse** — survive on what you built (food/water/sleep, power grid upkeep).

Tone: grim but not horror, brutalist/concrete aesthetic, military UI —
dark/gloomy/eerie, not bright. Bunker dimensions: width=24, depth=18
(`BunkerLayout.gd`). Solo dev project — keep everything modular/reusable,
avoid over-engineering.

For design-philosophy/feature-evaluation guidance (human/design reference
only, not part of the standard per-task doc chain), see `GAME_PHILOSOPHY.md`.

**Roadmap priorities (nothing currently in progress — confirm with Brannon
before starting any of these):**
- Water system Phase 2 — see `docs/systems/water/README.md` Known
  tradeoffs/Extension points: upgrade `WaterPipeDrawMode` to the full
  continuous-paint interaction model (Phase 1 shipped with the plan's own
  pre-approved segment-at-a-time fallback instead), then water purifiers/
  pumps/real consumption simulation/dashboard UI/T-splits.
- Main Menu scene, Death/game-over state — mentioned early in the project,
  never started, no trace in the repo currently.
- Placing `emergency_light` device instances near breaker boxes (device

Brannon. Phases 1/2/3/7 (renderer/environment baseline, GraphicsSettings

**Lighting blowout fix** (real bug hunt): 5 compounding settings fixed —
`tonemap_white` (never set → `5.0`), `tonemap_mode` (Filmic → true ACES `=3`),

`light_volumetric_fog_energy = 0.2`), `WallLight.LIGHT_ENERGY` (`4.5 → 2.0`),

`adjustment_brightness=0.87`, `adjustment_saturation=0.9`, `sdfgi_energy=0.7`.
**Pregen floor/wall retint:** `BunkerPregen._retint_meshlib()` duplicates
the MeshLibrary at runtime (never touches the shared binary `.meshlib`),
darkens floor/wall/pillar materials by the same factor Build Mode uses. Hit
`MeshLibrary.has_item()` doesn't exist — real API is `get_item_list() ->

spam (split `set_setting_live()`/`save_now()`), alarm vignette color clash
(changed to distinct hot amber), `apply_preset()` enum/int mismatch (retyped
to plain `int`).

## 9. Recurring gotcha — Godot class-cache / .uid staleness
A brand-new script with `class_name` (created via git, never opened/saved in
the Godot editor) sometimes isn't recognized ("Could not find type X") even
after a soft Project → Reload. Fix that's worked every time: **fully quit
Godot, delete the local `.godot/` cache folder (gitignored, safe,
auto-regenerates), reopen the project.**

`tools/godot_check.sh` (§16) catches the specific "Identifier X not declared"
symptom of this ahead of time for anything already pushed — run it before
handing code back to Brannon.

**New autoloads specifically:** `project.godot`'s `[autoload]` section is
owned by the Godot editor — hand-editing it via git while Godot has the
project open can get silently overwritten back to stale in-memory state on
next save/close. Brannon adds new autoloads himself via Project Settings >
Autoload in the editor (already done for `GraphicsSettings`, confirmed
working) — don't hand-edit that section in `project.godot` directly.

## 10. GDScript gotchas hit for real this project (don't repeat these)
- **`as` does NOT support enum casts** — only Object/class casts. Enums are
  plain ints in GDScript; assign/pass directly. Hit twice
  (`GraphicsSettings.gd`'s `msaa`/`_apply_to_viewport()`,
  `GraphicsSettingsPanel.gd`'s preset dropdown) — retype to plain `int`
  instead of fighting the cast when in doubt.
- **`MeshLibrary` has no `has_item()` method** — use
  `get_item_list() -> PackedInt32Array` and `.has(id)` on that.
- **Image-gen tool output has no real alpha channel** for particle/VFX
  textures needing transparency — bakes the transparency-checkerboard
  preview as opaque pixels. Build these with PIL directly instead.

## 11. Workflow rules — do not deviate
- **GitHub only.** Repo `Magellan36/bunker-game`, branch `main`. All work:
  read from repo → edit in sandbox → commit + push → Brannon pulls in Godot
  editor. **No zip downloads for this project, ever.**
- Use raw `git`/bash with a PAT (`ask_secrets` into `.env`, used as
  `https://x-access-token:$TOKEN@github.com/...`) — not the GitHub pipedream
  plugin (no delete-file action).
- Confirm root cause before writing a fix; one change at a time on complex
  systems (power grid especially); prompt Brannon to test + report back
  rather than stacking unverified changes. If several sessions fail to fix a
  bug, step back and reconsider the approach rather than repeating the same
  fix.
- Response format for code changes: Core Approach → Node Setup → Code → How
  it Works & Customization (loosen for small/obvious fixes).
- **No "god files":** don't let files accumulate too many unrelated
  responsibilities. Prefer a new, separate file for a self-contained new
  feature over bolting it onto an existing large file. Use the `_owner`
  back-reference pattern (see `docs/systems/power/README.md` §"Forbidden
  edits") if a new file needs to reach into an existing file's state without
  physically moving heavily-referenced dicts/vars.
- **Keep folders organized:** place new files in the subfolder matching
  their responsibility (§3), extending it with a new subfolder if nothing
  fits, rather than a flat/convenient location. Apply proactively at
  write-time, not as a later cleanup pass.
- **Debug logging: keep ALL of it, don't strip preemptively.** `[PM:*]`,
  `_pmdbg`, `_wdbg`, `[BreakerBox]`, `[MW:DIAG]`, `[SPLIT]` prints, plus
  `WIRE_DEBUG` toggles. F9 in-game dumps full wire/zone/PM state to
  `user://wire_debug.txt`. Only strip prints once Brannon explicitly asks
  for a given stable system.

## 12. Extraction methodology — reusable playbook for any future file split
1. Map every function's external dependencies first (grep for which
   dicts/vars/methods each candidate function touches) before deciding
   what's safe to move.
2. Never move a dict/state var referenced from 100+ scattered call sites in
   the same pass — keep it on the original owner, new class holds a plain
   `_owner: OriginalClass` back-reference, reaches in via `_owner._some_dict`.
3. Extract function bodies via verbatim line-range slicing (`sed -n
   'START,ENDp'`), never retype from memory.
4. Regex-substitute only the identified owner-side identifiers to
   `_owner.<name>` form — exclude ANY preceding `.`, not just `_owner.`.
5. Read the entire transformed output top-to-bottom before writing.
6. Keep identical-signature forwarding wrapper methods on the original class
   for every moved function still called from elsewhere (check signal
   `.connect()` bindings too, not just direct calls).
7. Reconcile function counts exactly as a final sanity check.
8. Bracket-balance-check every touched file before committing, OR (now
   preferred — see §16) run `tools/godot_check.sh` for a real parse check.
9. Commit each slice separately with a detailed message.
10. Verify zero external callers repo-wide for every moved function first.

## 16. Headless Godot compile check (July 2026)
`tools/godot_check.sh` runs the real Godot 4.6.3 engine headless
(`--headless --import` then `--headless --quit`) against the project to
catch script parse/type/compile errors and broken autoload/`class_name`
references BEFORE Brannon has to pull and hit them in his own editor. Run it
before handing back any code change. Auto-patches in the locally-registered
`GraphicsSettings` autoload for the run only (see §9), then restores
`project.godot` exactly afterward — never commits that patch.
Usage: `tools/godot_check.sh /path/to/Godot_v4.6.3-stable_linux.x86_64`
(binary lives in the sandbox at `/home/user/godot-bin/`, re-download if a
fresh sandbox doesn't have it — see `HANDOVER.md`).
Does NOT catch runtime/gameplay logic bugs or anything GPU-visual — still
needs Brannon's own in-editor test pass for those.

## 17. architecture.json (July 2026)
`tools/gen_architecture.py` — static text-scan of every `.gd` file, emits
`architecture.json` at repo root: per-script class_name/extends/exports/
signals/public functions/preloads/autoload references, approximate
dependency edges, and a circular-reference check. Regenerate with
`python3 tools/gen_architecture.py` at the start and end of a session.
Read this before opening source for a system's shape (dependencies,
signals, public API) — real behavior/logic still requires opening the
actual `.gd` file, this is structural only, not a real compile (that's what
`tools/godot_check.sh` is for).
